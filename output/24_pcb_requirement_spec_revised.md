# 24. 電子基板要求仕様書（改訂版）

## 1. 文書情報
- project: i-Power Engine
- version: 0.2.0
- document_type: pcb_requirement_spec
- status: draft-revised

## 2. 目的
本書は、HEMSコントローラを実現するための電子基板に対する要求を整理する。
改訂版では、通信プロファイルと性能予算を反映し、帯域・バッファ・更新領域・通信品質対応を踏まえた基板要求を定義する。

## 3. 基板に要求される機能
- 通信モジュール搭載（Wi-SUN / ネットワーク接続）
- 制御・収集処理を行うCPU/MCU
- 不揮発性保存領域
- OTA更新を支える構成
- 保守・デバッグ導線
- 電源安定化
- 異常時の安全側動作を支える設計

## 4. 通信入力 / 内部通信の前提
### 4.1 Bルート入力
- 低速
- 通信品質が不安定
- ポーリング主体
- 欠損許容設計が必要
- 基板側では通信断続を前提に安定動作すること

### 4.2 ECHONET Lite
- 取得/制御対象が機器差異を持つ
- request/response主体
- 応答遅延や失敗を吸収できること
- 制御系コマンドは順序性を維持すること

### 4.3 端末→クラウド同期
- TLS / 相互認証前提
- 時系列同期とイベント優先送信を両立すること
- クラウド断時のローカル保持を支えること

## 5. 性能予算から導くハード要求
### 5.1 バッファ / メモリ
- 時系列同期用バッファ: 最低 64KB クラス
- イベント優先キュー: 最低 8KB クラス
- Bルート / ECHONET受信処理を同時保持できるRAM余裕
- 推定 / 欠損判定、暗号処理、再送制御を含めても安全縮退が止まらないこと

### 5.2 不揮発領域
- 設定保存領域
- ログ保存領域
- 未送信データ保持領域
- OTA更新一時領域
- 更新状態管理領域
- 合計として少なくとも数MB級の設計余裕を持つこと

### 5.3 OTA
- firmware_package の peak_message_size は最大 2MB級
- 一時領域は最低 4MB級を前提
- 整合性検証とロールバックのため、現行領域と更新領域を分離できる構成が望ましい

## 6. CPU/MCU 要求観点
- 通信処理、データ収集、再送、暗号処理、同期、ログ処理を並行実行可能なこと
- 更新中でも watchdog / 安全縮退を維持可能なこと
- イベント優先処理により通常同期を圧迫しすぎないこと

## 7. インターフェース要求
- Wi-SUN系接続
- ネットワーク接続（クラウド同期）
- 保守 / デバッグ用インターフェース
- 必要に応じた将来拡張用I/O余地
- 電源 / リセット / 復旧導線

## 8. 安全・保守・更新要求
- 通信断時継続性を支えること
- 機器・クラウド間セキュリティ実装を支えること
- OTA更新失敗時ロールバックを支えること
- 承認操作 / 制御系の監査ログ保存を支えること
- 更新失敗や断電時にも危険側へ遷移しないこと

## 9. リスク・留意事項
- 機器差異により取得できるデータが揺れる
- OTA更新失敗で端末が不安定化する
- 遠隔制御経路のセキュリティ不足
- 通信品質悪化時に再送やキューが過剰増加する可能性がある

## 10. 追加で詰めるべき項目
- 電源仕様
- 温度 / 環境条件
- EMC / ノイズ要件
- コネクタ構成
- 実装サイズ制約
- 量産試験・検査観点
- 法規 / 認証条件
- MCU / SoC候補選定
- RAM / Flash / ストレージ容量の確定
