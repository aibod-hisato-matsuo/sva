# 01. プロダクト概要

## プロジェクト情報
- プロジェクト名: i-Power Engine
- バージョン: 0.1.0
- ステータス: draft
- 言語: ja
- 最終更新日: 2026-03-17

## ビジョン
メーカー非依存で太陽光発電および蓄電池の見える化・選択・制御を可能にする次世代HEMSを実現する。

## 目的
- 太陽光・蓄電池・買電/売電データを安定取得する
- ユーザーが自らの意思でエネルギー運用方針を選択できるようにする
- 実測値と推定値を明確に区別して提示する
- 機器差異を吸収して将来の対応機種拡張を容易にする
- 通信断や異常時にも安全側へ縮退する

## スコープ
- HEMSコントローラ
- クラウドバックエンド
- 家庭内ユーザーアプリ
- 管理者アプリ
- 太陽光発電設備
- 蓄電池設備
- スマートメーター

## スコープ外
- 初期フェーズにおける全メーカー完全対応
- 管理者アプリからの直接的な遠隔制御
- 家庭内の全家電を対象にした包括HEMS

## ユーザーモード
### 見える化モード
- ID: mode_visualize
- 説明: 計測値と推定値を可視化し、ユーザーの理解を支援する。
- ユーザー承認必須: いいえ
- 自動化レベル: visualization

### 提案モード
- ID: mode_recommend
- 説明: 節約優先や備え優先などの方針に応じた推奨を提示する。
- ユーザー承認必須: はい
- 自動化レベル: recommendation

### 自動実行モード
- ID: mode_auto
- 説明: ユーザーが事前承認した方針に基づき、自動制御を行う。
- ユーザー承認必須: はい
- 自動化レベル: automatic_execution


# 02. SVAレイヤー設計

## L0: Purpose / Policy
- 意図: 家庭エネルギー主権の実装
- 主要要素:
- ユーザー選択可能性
- メーカー非依存
- 安全側縮退
- 提案主体・代行しないAI
- 制約:
- AIは意思決定を代行しない
- 誤解を招く表示を避ける

## L1: Requirement / Schema
- 意図: 共通意味モデルと要求の定義
- 主要要素:
- 計測値
- イベント
- ユーザーモード
- 推定値フラグ
- 機器能力宣言
- 制約:
- 実測値と推定値を区別できること
- メーカー名ではなく能力差で表現すること

## L2: Logic / AI
- 意図: 提案ロジックと制御ルールの定義
- 主要要素:
- 節約提案
- 備え優先提案
- 売電/自家消費バランス
- 異常検知・保護
- 制約:
- 安全制御は端末側へ依存可能な範囲を限定する
- 提案根拠を説明可能にする

## L3: Orchestration
- 意図: Edge/Cloud/Appの責任分割と同期
- 主要要素:
- データ同期
- 通知フロー
- 障害時縮退
- 承認フロー
- 制約:
- 通信断時も最低限の運転継続
- クラウド障害で危険側に倒れない

## L4: Interface / Adapter
- 意図: Wi-SUN/ECHONET Lite/メーカー差異の吸収
- 主要要素:
- 取得チャネル統一
- プロパティマップ
- コマンド変換
- エラー標準化
- 制約:
- アプリ層にメーカー固有差分を漏らさない
- 追加メーカーを後付けしやすくする

## L5: Execution / Physical
- 意図: 実機接続と物理安全実行
- 主要要素:
- HEMS端末
- PV
- 蓄電池
- スマートメーター
- 家庭内通信
- 制約:
- 端末は安全実行主体
- 通信断時の保守動作を持つ

# 04. 責任分割

## HEMSコントローラ
- 主責務:
- 機器接続維持
- メーカー差異吸収
- ローカルでの安全実行
- 通信断時の最低限動作
- 計測値取得と一部推定補助
- やってはいけないこと:
- 重い最適化判断を常時担当しない
- 家庭内UIロジックを持たない
- 障害時挙動:
- クラウド断時は保守運転へ移行する
- 未取得値は欠損として扱い、無理な制御を行わない

## クラウド
- 主責務:
- データ蓄積
- 可視化用集計
- 提案生成
- ユーザーモード管理
- 異常検知補助
- 外部API連携
- やってはいけないこと:
- 通信断前提の安全制御に依存しない
- メーカー固有プロトコルを直接扱わない
- 障害時挙動:
- 端末側の継続動作を前提に、復旧後に差分同期する
- 推奨機能停止時も可視化機能を優先維持する

## 家庭内アプリ
- 主責務:
- 状態把握
- モード設定
- 提案確認
- 承認操作
- 履歴参照
- やってはいけないこと:
- 実機プロトコルを直接扱わない
- 実測と推定を混同させる表示をしない
- 障害時挙動:
- 最新同期時刻を明示する
- 未更新時はデータ鮮度不足を表示する

## 管理者アプリ
- 主責務:
- 運用監視
- 接続状態把握
- 障害分析
- サポート支援
- やってはいけないこと:
- 原則として直接制御しない
- ユーザーの意思決定を代替しない
- 障害時挙動:
- 障害分類ごとに優先度表示を行う
- サポート向けに取得不可項目を明示する

# 06. ユースケース

## ユーザーが家庭エネルギー状態を確認する
- ID: uc_001
- ゴール: PV・蓄電池・買電/売電の現在値と推移を理解する
- アクター: user, user_app, cloud, edge_controller, smart_meter, pv_system, battery_system
- 前提条件:
- HEMS端末が機器と接続済みである
- 少なくとも1つ以上の計測チャネルが有効である
- メインフロー:
1. [edge_controller] スマートメーター・PV・蓄電池からデータを収集する
2. [cloud] 取得データを正規化・蓄積する
3. [cloud] 欠損がある場合は推定値を生成しフラグ付与する
4. [user_app] 現在値・推移・推定フラグを表示する
- 代替フロー:
#### 条件: 一部データが未取得
- 推定値へ切替
- 最新同期時刻を表示
- 未取得項目を注記する
- 事後条件:
- ユーザーは実測/推定を区別して状態把握できる
- データは履歴として参照可能である
- 関連レイヤー: L1, L3, L4, L5
- 関連リスク: r_001, r_002

## ユーザーが節約提案を確認し設定する
- ID: uc_002
- ゴール: 契約料金・発電状況・蓄電池状態に応じた推奨を受け取り、モードを選択する
- アクター: user, user_app, cloud, external_api, battery_system
- 前提条件:
- ユーザー契約情報が登録されている
- 料金関連情報または既知の料金テーブルが利用可能である
- メインフロー:
1. [cloud] 契約料金、消費履歴、発電・蓄電池状態を参照して提案を生成する
2. [user_app] 提案根拠と予想効果を表示する
3. [user] 節約優先などのモードを選択する
4. [cloud] モードを保存し必要に応じて端末へ配信する
- 代替フロー:
#### 条件: 料金APIが取得できない
- 直近既知情報で暫定提案を生成
- 推定扱いを表示する
- 事後条件:
- ユーザー方針が保存される
- 次回以降の提案・制御に反映される
- 関連レイヤー: L0, L1, L2, L3
- 関連リスク: r_002, r_004

## 停電リスク時に備え優先へ移行する
- ID: uc_003
- ゴール: 停電・災害リスクがある場合に蓄電池残量確保を優先する
- アクター: cloud, edge_controller, user_app, battery_system, external_api
- 前提条件:
- 備え優先ポリシーが有効化可能である
- 蓄電池制御機能が利用可能である
- メインフロー:
1. [external_api] 停電警報または災害情報を提供する
2. [cloud] 備え優先ルールを評価する
3. [edge_controller] 必要に応じて蓄電池の放電抑制またはSOC下限制御を行う
4. [user_app] 備え優先への移行と理由を通知する
- 代替フロー:
#### 条件: クラウド接続なし
- 端末の事前設定ルールにより保守動作
- 復旧後に状態同期する
- 事後条件:
- 備え優先の運転方針が適用される
- ユーザーは変更理由を理解できる
- 関連レイヤー: L0, L2, L3, L4, L5
- 関連リスク: r_001, r_005

## 通信断時に安全側へ縮退する
- ID: uc_004
- ゴール: 通信断や機器異常が発生しても危険側に倒れない
- アクター: edge_controller, cloud, admin_app, user_app
- 前提条件:
- 端末に最低限の保守ルールが設定されている
- メインフロー:
1. [edge_controller] 通信断または機器応答異常を検知する
2. [edge_controller] 安全側縮退ルールを適用する
3. [cloud] 接続復旧後に差分同期を行う
4. [admin_app] 障害内容と影響範囲を表示する
- 代替フロー:
#### 条件: ユーザーアプリ未接続
- 管理者向け監視のみ優先
- 後続同期時に履歴通知する
- 事後条件:
- 危険な制御は抑止される
- 障害経緯が記録される
- 関連レイヤー: L3, L4, L5
- 関連リスク: r_001, r_003, r_005


# 07. 非機能要件とリスク

## 非機能要件
### 通信断時継続性
- ID: nfr_001
- 区分: availability
- 要求: クラウド接続断が発生しても端末は最低限の安全動作を継続できること。
- 目標値: 主要安全機能はローカル継続
- 検証方法: ネットワーク遮断試験
- 関連コンポーネント: edge_controller, network, cloud

### 実測/推定区分
- ID: nfr_002
- 区分: data_quality
- 要求: 推定値は必ずフラグを持ち、UI上で実測値と明確に区別されること。
- 目標値: 100%識別可能
- 検証方法: UIレビューおよびAPI検証
- 関連コンポーネント: cloud, user_app, admin_app

### 機器・クラウド間セキュリティ
- ID: nfr_003
- 区分: security
- 要求: 端末・クラウド間の通信は認証され、改ざん耐性を持つこと。
- 目標値: 相互認証および暗号化通信
- 検証方法: セキュリティレビュー・ペネトレーション試験
- 関連コンポーネント: edge_controller, cloud, network

### 提案の説明可能性
- ID: nfr_004
- 区分: ux
- 要求: 提案表示には根拠と期待効果を併記し、ユーザーが理由を理解できること。
- 目標値: 主要提案100%に根拠表示
- 検証方法: UXレビュー
- 関連コンポーネント: cloud, user_app

### OTA更新の安全性
- ID: nfr_005
- 区分: ota
- 要求: OTA更新は失敗時にロールバック可能であり、更新中に危険側へ遷移しないこと。
- 目標値: 失敗時ロールバック
- 検証方法: OTA障害注入試験
- 関連コンポーネント: edge_controller, cloud

## リスク登録簿
