# 23. ファームウェア仕様書（改訂版）

## 1. 文書情報
- project: i-Power Engine
- version: 0.2.0
- document_type: firmware_spec
- status: draft-revised

## 2. 目的
ファームウェアは、HEMSコントローラ内部でのタスク実行主体として、通信処理、データ収集、ローカルルール適用、安全縮退、同期、更新処理を実装する。
改訂版では、通信プロファイルと性能予算を反映し、タスク設計とバッファ管理の要件を明確化する。

## 3. 外部要求との対応
- プロトコル吸収（Wi-SUN / ECHONET Lite）
- ローカル安全縮退
- OTA更新
- 時系列同期
- イベント優先送信

## 4. 想定ソフトウェア構成
- 通信タスク（Wi-SUN Bルート）
- 通信タスク（ECHONET Lite）
- データ正規化タスク
- 欠損 / stale / estimated 判定タスク
- クラウド同期タスク
- イベント優先送信タスク
- ローカルルール / 安全縮退タスク
- ログ / 診断タスク
- OTA更新タスク

## 5. 通信タスク要求
### 5.1 Bルート受信タスク
- poll周期: 30s-300s の範囲で設定可能
- timeout_ms: 5000
- retry_policy: 最大3回再試行
- 失敗時処理: 欠損マーク付与、通常処理継続
- バッファ方針: 最新値優先、古い取得待ちは破棄

### 5.2 ECHONET Lite タスク
- 取得周期: 10s-60s の範囲で設定可能
- timeout_ms: 3000
- retry_policy: 取得系2回、制御系1回
- 失敗時処理: stale / estimated 化、制御失敗時は制御停止
- バッファ方針: 制御系順序維持、計測系最新値優先

### 5.3 クラウド同期タスク
- batch_sync周期: 10s-60s
- max_payload_size_bytes: 4096
- 送信失敗時: ローカルキュー保持 + 指数バックオフ
- イベントは優先送信対象
- 復旧後: 差分再同期

## 6. 性能予算反映
### 6.1 計測データ処理
- Bルート計測フロー: 2KBバッファ、1時間保持
- PV/蓄電池状態フロー: 4KBバッファ、30分保持
- 欠損許容と stale 判定をタスク内で扱う

### 6.2 同期・イベント
- 時系列同期フロー: 64KB級バッファ前提
- イベント/異常通知: 8KB優先キュー
- イベントは計測バッチより優先順位を上げる

### 6.3 OTA
- OTA更新フロー: 4MB級一時領域を前提
- 分割転送、整合性検証、ロールバックが必須
- 更新中は安全縮退処理を阻害しないスケジューリングが必要

## 7. 状態遷移の考え方
- 起動

- 更新中

## 8. バッファ / キュー設計ガイド
- Bルート受信バッファ: 小容量、最新値優先
- ECHONET受信バッファ: プロパティ差異を吸収できるサイズ
- クラウド送信キュー: 時系列データ用の大容量キュー
- イベントキュー: 別系統の高優先キュー
- OTA一時領域: 通常バッファと分離
- 永続化対象: 設定、未送信データ、監査ログ、更新状態
